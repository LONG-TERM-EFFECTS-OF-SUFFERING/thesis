# 03 - Create research project

Story: US-004, Create research project.

Status note: this document explains the final reviewed implementation after the US-004 code review fixes.

## What this story added

This story adds the first domain feature for OpenTube Insights: research projects.

A researcher can now:

- create a project with a name, description and default language.

- see saved projects in the frontend.

- use a backend API that stores projects in the database.

The project record is the parent object for the next story, saved YouTube search queries.

## Why this matters

YouTube collection needs research context. A saved query, collection run or result should not float by itself; it should belong to a study topic.

The `ResearchProject` model provides that study workspace. It stores ownership, status and timestamps so later data can be traced back to the research topic that created it.

## File-by-file explanation

### `src/api/core/models.py`

This file defines `ResearchProject`.

Important fields:

- `id`: a UUID primary key generated by Django.

- `owner`: a link to Django's user model.

- `name`: the project name.

- `description`: optional project details.

- `status`: either `active` or `archived`.

- `default_language`: optional language hint, such as `es`.

- `created_at` and `updated_at`: timestamps managed by Django.

The model also has two database constraints:

- one owner cannot have two projects with the same name.

- `status` must be one of the allowed values.

### `src/api/core/migrations/0001_initial.py`

This migration creates the `research_projects` database table.

Run migrations whenever a Django model changes:

```bash

```

### `src/api/core/serializers.py`

The serializer controls how project records become JSON and how incoming JSON is validated.

The create form submits `name`, `description` and `default_language`. The API also supports status updates for future archive workflows, but the browser cannot choose `id`, `owner_id`, `created_at` or `updated_at`.

The serializer also checks duplicate project names for the same owner before the database constraint is hit, which gives the frontend a clearer validation error. The create view still catches the database constraint path so a near-simultaneous duplicate create returns a validation response instead of a server error.

### `src/api/core/views.py`

This file now has project API views in addition to the health endpoint.

Project ownership works like this:

- if the request has an authenticated user, that user owns the project.

- if the request is unauthenticated, local development uses `OPENTUBE_DEFAULT_OWNER_USERNAME`, defaulting to `local-researcher`.

That fallback keeps the local thesis workflow usable before a future authentication story adds sign-in.

### `src/api/core/urls.py`

The project API routes are:

```text
GET /api/projects/
POST /api/projects/
GET /api/projects/<uuid>/
PATCH /api/projects/<uuid>/
```

The existing health route remains:

```text
GET /api/health/
```

### `src/ui/src/apiClient.ts`

The frontend API client now includes typed project helpers:

- `listResearchProjects`.

- `createResearchProject`.

- `updateResearchProject`.

It validates unknown JSON before React uses it. This keeps API shape assumptions in one boundary module instead of scattering them through components.

The validation is endpoint-specific: list requests must return a list, while create and update requests must return one project. When the backend returns field validation errors, the client keeps those details so the form can show a useful message.

### `src/ui/src/projectState.ts`

This file contains project-list state helpers. It keeps the list update logic testable outside React.

`mergeLoadedProjects` protects a small request-ordering edge case: if the first list request finishes after a user creates a project, the older list response must not hide the newly created row.

### `src/ui/src/App.tsx`

The first screen is now a project workspace.

It loads saved projects, shows empty/loading/error states, creates a project from the form and keeps the backend health state visible in the header. When a project is created, it is added to the visible list immediately without waiting for a page reload.

### `src/ui/src/App.css`

This file styles the project workspace with a two-column desktop layout and a one-column mobile layout.

### `src/ui/tests/projectApi.test.ts`

These tests cover project API helpers, JSON shape validation and fetch request details.

### `src/ui/tests/projectState.test.ts`

These tests cover project-list merging so stale list responses do not hide newly created projects.

### `src/docker-compose.yml`, `src/.env.example`, `src/api/.env.example`

These files expose `OPENTUBE_DEFAULT_OWNER_USERNAME` for local Docker and backend runs.

## How the pieces connect

```text
Project form
  -> src/ui/src/App.tsx
  -> createResearchProject in src/ui/src/apiClient.ts
  -> POST /api/projects/
  -> ResearchProjectListCreateView
  -> ResearchProjectSerializer
  -> ResearchProject model
  -> research_projects table
```

When the frontend loads, it also calls:

```text
GET /api/projects/
```

That returns projects for the current effective owner.

## Common errors

### Duplicate project name

If the same local owner already has a project with that name, the API returns a validation error. Use a different project name or archive/update the old project later when that workflow exists.

The frontend shows the backend field message near the form, so duplicate-name errors should be understandable without opening browser developer tools.

### Unexpected owner

Local unauthenticated requests use `local-researcher` by default. Set `OPENTUBE_DEFAULT_OWNER_USERNAME` if you need a different local owner bucket.

## Key concepts

- A serializer validates and converts between JSON and Django model data.

- A local fallback owner is temporary glue until authentication is implemented.
