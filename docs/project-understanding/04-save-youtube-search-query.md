# 04 - Save YouTube search query

Story: US-005, Save YouTube search query.

Status note: this document explains the final reviewed US-005 implementation after the saved-query review fixes.

## What this story added

This story lets a researcher save reusable YouTube search criteria inside an existing research project.

A researcher can now:

- choose an active project in the frontend.

- save a manual YouTube search query with a name and search term.

- add optional prompt text, date bounds, a video discovery limit, region, language and notes.

- see saved queries for the selected project after creation and after page reload.

- use backend API endpoints that keep saved queries scoped to the effective project owner.

This story does not call the YouTube API yet. It stores the criteria that later stories will use to start audited collection runs.

## Why this matters

YouTube collection needs reusable and auditable input criteria. If a researcher only types search terms directly into a collection run, it becomes harder to prove what was requested, who created it and which project it belonged to.

US-005 answers this question:

```text
Can the app save YouTube search criteria as project-owned records before running collection?
```

The answer is now yes.

## File-by-file explanation

### `src/api/core/models.py`

This file defines `SavedQuery`.

Important fields:

- `id`: a UUID primary key generated by Django.

- `project`: the `ResearchProject` that owns the query.

- `created_by`: the user who saved the query.

- `name`: the researcher-facing query name.

- `natural_language_prompt`: optional text describing the research intent.

- `query_source`: currently defaults to `manual`; `llm_generated` is reserved for later LLM query translation stories.

- `search_term`: the required keyword or phrase for the YouTube search.

- `published_after` and `published_before`: optional publication date bounds.

- `max_videos_to_discover`: optional total number of videos the saved query should try to discover later.

- `region_code` and `relevance_language`: optional YouTube search filters.

- `structured_query_params`: JSON parameters that mirror the YouTube search request shape.

- `llm_model_name`, `llm_prompt_version` and `translation_confidence`: reserved metadata for later LLM-generated queries.

- `notes`: optional researcher notes.

- `created_at` and `updated_at`: timestamps managed by Django.

The model has one uniqueness constraint:

```text
project + created_by + name must be unique
```

That means the same user cannot create two saved queries with the same name in the same project. The same name can still exist in another project.

### `src/api/core/serializers.py`

`SavedQuerySerializer` controls how saved-query records become JSON and how incoming JSON is validated.

The browser can send fields such as `name`, `search_term`, date bounds, filters, structured parameters and notes. It cannot choose `id`, `project`, creator username, source metadata or timestamps.

The serializer validates that:

- the query name is not blank.

- the search term is not blank.

- `max_videos_to_discover` is at least `1` when provided.

- `structured_query_params` is a JSON object.

- `published_before` is not earlier than `published_after`.

- the query name is not duplicated for the same project and creator.

### `src/api/core/views.py`

`SavedQueryListCreateView` adds the project-scoped saved-query API.

It uses the existing effective-owner behavior from US-004:

- authenticated requests use the authenticated user.

- unauthenticated local requests use the fallback owner from `OPENTUBE_DEFAULT_OWNER_USERNAME`, defaulting to `local-researcher`.

The view looks up the project by both project id and owner. If another user tries to access a project they do not own, the endpoint returns `404` instead of exposing the project or its queries.

On create, the view derives:

```text
project    -> from the URL
created_by -> from the effective owner
```

The client does not get to choose either value.

The view also catches duplicate-name database races and returns a validation error instead of a server error.

### `src/api/core/urls.py`

The saved-query route is:

```text
GET  /api/projects/<project_id>/queries/
POST /api/projects/<project_id>/queries/
```

The route sits under a project because saved queries should never float outside research context.

### `src/ui/src/apiClient.ts`

The frontend API boundary now includes:

- `SavedQuery`.

- `SavedQueryInput`.

- `isSavedQuery`.

- `listSavedQueries(projectId)`.

- `createSavedQuery(projectId, input)`.

These helpers keep backend URL construction centralized through `buildApiUrl`. React components do not hardcode backend hostnames.

The client also validates unknown JSON before the UI uses it. If the backend response shape changes unexpectedly, the error is caught at the API boundary.

### `src/ui/src/savedQueryState.ts`

This file contains the saved-query list merge helper.

It protects a request-ordering edge case:

```text
1. The UI starts loading queries for a project.
2. The user saves a new query.
3. The older list request finishes after the create request.
```

Without the merge helper, the older list response could hide the newly created query. The helper preserves queries already shown in the UI while still accepting loaded queries from the backend.

### `src/ui/src/App.tsx`

The app remains one working research workspace screen.

It now has two main areas:

- Projects: create a project and choose the active project.

- Saved queries: save and list manual queries for the active project.

The saved-query form uses native controls:

- text inputs for name, search term, region and language.

- `datetime-local` inputs for publication date bounds.

- a number input for `max_videos_to_discover`.

- text areas for the natural-language prompt and notes.

When the user saves a query, the frontend sends structured query parameters like:

```json
{
    "part": "snippet",
    "type": "video",
    "q": "movilidad Cali"
}
```

That gives later collection stories a clear starting shape for YouTube `search.list` parameters.

### `src/ui/src/App.css`

The styles keep the dense workspace layout from US-004 and add saved-query cards.

Each query card shows the query name, search term, source, video discovery limit, region, language and date bounds.

## How the pieces connect

```text
Saved-query form
  -> src/ui/src/App.tsx
  -> buildSavedQueryInput
  -> createSavedQuery in src/ui/src/apiClient.ts
  -> POST /api/projects/<project_id>/queries/
  -> SavedQueryListCreateView
  -> SavedQuerySerializer
  -> SavedQuery model
  -> saved_queries table
```

When the active project changes, the frontend calls:

```text
GET /api/projects/<project_id>/queries/
```

That returns only saved queries for the selected project and effective owner.

## Commands

Create a project first:

```bash
curl -sS -X POST "http://127.0.0.1:${API_HOST_PORT:-8000}/api/projects/" \
  -H "Content-Type: application/json" \
  -d '{"name":"Cali mobility study","description":"Local thesis test","default_language":"es"}'
```

Use the returned project `id` to create a saved query:

```bash
curl -sS -X POST "http://127.0.0.1:${API_HOST_PORT:-8000}/api/projects/<project-id>/queries/" \
  -H "Content-Type: application/json" \
  -d '{
    "name":"Cali mobility videos",
    "natural_language_prompt":"Videos about mobility debates in Cali",
    "search_term":"movilidad Cali",
    "published_after":"2026-01-01T00:00:00Z",
    "published_before":"2026-06-01T00:00:00Z",
    "max_videos_to_discover":250,
    "region_code":"CO",
    "relevance_language":"es",
    "structured_query_params":{"part":"snippet","type":"video","q":"movilidad Cali"},
    "notes":"Manual seed query for sprint testing."
  }'
```

List saved queries for a project:

```bash
curl -sS "http://127.0.0.1:${API_HOST_PORT:-8000}/api/projects/<project-id>/queries/"
```

## Common errors

### No active project

A saved query must belong to a project. Create or select a project before saving query criteria.

### Missing name or search term

The API rejects blank query names and blank search terms. These are the minimum fields needed to identify and later run a query.

### Duplicate query name

If the same creator already has a saved query with that name in the same project, the API returns a validation error. Use a different query name.

### Invalid date range

`published_before` must be the same as or later than `published_after`.

### Invalid video discovery limit

`max_videos_to_discover` must be at least `1`. It is the total number of videos the query should try to discover, not the YouTube API's per-request `maxResults` page size.

### Unexpected owner

Local unauthenticated requests use `local-researcher` by default. Set `OPENTUBE_DEFAULT_OWNER_USERNAME` if you need a different local owner bucket.

## Key concepts

- A saved query is stored criteria, not a collection run.

- Project-scoped URLs make ownership and research context explicit.

- The backend derives trusted relationship fields from the URL and owner instead of trusting client JSON.

- JSON fields are useful for structured API parameters that may grow over time.

- Native form controls are enough for this story; no new frontend dependency is needed.
